# CUMCM 全流程工作流

用于 CUMCM 及同类中文数学建模竞赛的端到端协作：题目与附件核验、建模方案比较、代码与数据处理、结果验证、论文材料整理和提交前审计。

## 启动方式

在对话中输入：

```text
使用 $cumcm-workflow 分析这个题目。题目文件在……，附件在……
```

项目开始前必须提供：完整题目、题目声明的全部附件（或明确“无附件”）、工作目录，以及当届竞赛/格式/学校规则。历史题缺少对应年份规则时，产物只能标为练习稿或复核稿，不能声称投稿合规。

## 产物结构

```text
项目名/
├── 00_题目与计划/       # 任务看板、变更记录、待补材料、题目理解
├── 01_数据/             # 原始数据、清洗数据、外部数据登记
├── 02_模型/             # 分析、假设、候选方案、主模型、验证局限
├── 03_代码/             # 可复现程序与运行说明
├── 04_结果与图表/       # 已验证的结果表和图表
├── 05_论文材料/         # 论文源文件、PDF、发布审计元数据
└── LOG.md               # 全程时间线
```

可执行以下命令创建标准目录：

```bash
python3 scripts/init_cumcm_project.py /path/to/project
```

## 核心原则

- 先核验题意、附件和规则，再建模或写结果。
- 不编造数据、引用、实验、结论或论文内容。
- 对关键子问题至少给出两条可行路线；由用户选择最终模型。
- 既有思路或代码存在重大逻辑缺陷时，必须指出并征求是否重建。
- 每个可运行程序至少实际执行一次；启发式优化只可称为可行候选，除非证明全局最优。
- 用户拥有最终决定权；团队协作以任务看板、变更记录和 `LOG.md` 为准，也支持单人使用。

## 论文发布门槛

在 `05_论文材料/` 维护：

- `论文元数据.json`：题目年份、规则年份、论文类别、模板分类、模型版本、审计范围。
- `证据清单.json`：每个关键结论对应的论文位置、模型文件、代码、运行记录和数据/输出文件。

提交 PDF 前运行：

```bash
python3 scripts/audit_paper_evidence.py /path/to/project
```

审计失败时不得交付最终版。它会拦截规则年份不匹配、模板误用、模型版本混用、未运行代码、失效附录路径、缺失证据链和遗留术语等问题。

详细字段说明见 [references/paper-release-gate.md](references/paper-release-gate.md)。
